Introduction

What is APOC?

APOC — short for Awesome Procedures on Cypher — is the most widely-used extension library for the Neo4j graph database. Originally standing for "A Package Of Components" when it was first bundled with Neo4j back in 2009, APOC has grown into a comprehensive toolkit of 450+ procedures and functions that dramatically extend what is possible with plain Cypher queries.

At its core, APOC fills the gaps between what the Cypher query language can express natively and what real-world production workloads demand. Need to load data from a REST API in a single Cypher statement? Perform bulk graph refactoring across millions of nodes without writing Java? Run periodic background jobs, integrate with relational databases, or generate and store vector embeddings directly in Neo4j? APOC has a procedure for all of it.

APOC procedures are written in Java, deployed as a plugin JAR, and called from Cypher using the familiar CALL apoc.namespace.procedureName(args) syntax. Functions — a subset of APOC — can be embedded directly in expressions and predicates just like built-in Cypher functions such as toLower() or size().

💡
Etymology Trivia "Apoc" was also the name of the technician and driver aboard the Nebuchadnezzar in the Matrix — he was killed by Cypher. The Neo4j community loved the pun: APOC fills in everything that Cypher cannot do alone.
Architecture

APOC Core vs. APOC Extended

As of Neo4j 5.0, APOC was split into two separate modules to improve stability, security, and availability on managed cloud services:

APOC Core

The officially supported module maintained by Neo4j. It contains 450+ procedures and functions covering data import/export, graph refactoring, Cypher execution, data structures, temporal utilities, virtual graphs, and more. APOC Core is available on Neo4j AuraDB, Neo4j Sandbox, Docker, and Neo4j Desktop with a single click.

APOC Extended

A community-maintained module with ~50 additional procedures that have external dependencies (e.g. MongoDB, ElasticSearch, Couchbase connectors), experimental features, or NLP integrations. APOC Extended is not available on managed cloud services like Aura — it requires a self-hosted Neo4j instance. Procedures are called with the same apoc.* prefix.

🔒
Security Note Some APOC procedures — particularly those that read files from disk, open network connections, or run arbitrary Cypher — are restricted by default. They must be explicitly enabled in neo4j.conf (e.g. apoc.import.file.enabled=true) before they can be called.
Setup

Installation & Configuration

APOC Core can be installed in four ways, depending on your environment:

Neo4j Desktop (Recommended for Development)

Open your database → click Manage → go to the Plugins tab → click Install next to APOC. Restart the database. No configuration required.

Docker

# Pass the NEO4J_PLUGINS environment variable — Neo4j will # automatically download and configure APOC at container startup. docker run -p 7474:7474 -p 7687:7687 \ -e NEO4J_PLUGINS='["apoc"]' \ -e NEO4J_apoc_export_file_enabled=true \ -e NEO4J_apoc_import_file_enabled=true \ -e NEO4J_apoc_import_file_use__neo4j__config=true \ neo4j:latest

Self-Hosted / Manual

Download the apoc-<version>-core.jar from the GitHub releases page (version must match your Neo4j major.minor version), place it in $NEO4J_HOME/plugins/, and restart Neo4j.

AuraDB

APOC Core is available by default in all Neo4j Aura instances. No installation is needed — just start calling apoc.* procedures. Some restricted procedures are not available on Aura for security reasons.

Verify Installation

// Verify APOC is loaded and check your version RETURN apoc.version() AS apocVersion; // List all available APOC procedures CALL apoc.help('apoc') YIELD name, description, signature RETURN name, description LIMIT 20; // Search for procedures in a specific namespace CALL apoc.help('apoc.load') YIELD name, description RETURN name, description;

Category 01
Import
apoc.load.*

The Import category is one of the most heavily used in practice. It enables Neo4j to consume external data sources directly inside Cypher queries — without any application code acting as an intermediary. You can stream JSON from REST APIs, ingest CSV exports from spreadsheets or data warehouses, parse XML feeds, query relational databases via JDBC, and even walk local file directories. Combined with apoc.periodic.iterate, Import procedures become the backbone of production ETL pipelines capable of loading tens of millions of records efficiently.

ETL Pipelines REST API Integration CSV Ingestion JDBC / RDBMS Migration XML Parsing
ProcedureDescriptionWhen to Use
apoc.load.jsonStreams a JSON document from a URL or local file path. Yields each top-level element as a value map. Supports HTTP headers and query params.Loading product catalogs, API responses, or JSONL files into the graph
apoc.load.jsonArrayLoads a JSON array and yields each element individually, ideal when the root of the document is an array rather than an object.Batch-processing arrays of records from JSON exports
apoc.load.csvStreams a CSV file row by row. Supports custom delimiters, quote characters, null values, and automatic type coercion for each column.Importing spreadsheet exports, data warehouse dumps, or CRM data
apoc.load.xmlParses an XML document into a nested map structure. Handles attributes, namespaces, and CDATA sections. Yields the root element as a traversable map.Ingesting XML data feeds, SOAP responses, or configuration files
apoc.load.jdbcExecutes a SQL query against any JDBC-compliant database (PostgreSQL, MySQL, Oracle, SQL Server) and streams the result set into Cypher.Migrating relational data to Neo4j or enriching graph nodes with RDBMS data
apoc.load.jsonParamsLike apoc.load.json but accepts custom HTTP method, headers, and a payload body — enabling POST requests to authenticated REST APIs.Calling authenticated APIs with bearer tokens or API keys
apoc.load.directoryLists files in a Neo4j import directory matching a glob pattern. Returns file name, path, size, and last-modified timestamp.Building directory-based file ingestion loops; discovering available import files

Example — Load from a paginated JSON API:

// Stream a JSON file from a URL and create nodes for each record. // apoc.load.json yields one `value` map per top-level object. WITH 'https://api.example.com/products.json' AS url CALL apoc.load.json(url) YIELD value AS row MERGE (p:Product {sku: row.sku}) ON CREATE SET p.name = row.name, p.price = toFloat(row.price), p.brand = row.brand RETURN count(p) AS productsLoaded;

Category 02
Export
apoc.export.*

The Export category provides a snapshot and serialisation capability for your graph data. You can export the entire graph, a specific subgraph defined by a query, or a set of nodes and relationships to formats including JSON, CSV, GraphML, Cypher scripts, and Gephi-compatible files. These exports serve as backups, graph migration artifacts, ML training datasets, or embedding pipelines. The stream: true config option returns data as a text column instead of writing a file — useful for sending graph content to an external process in real time.

Backups Data Migration ML Dataset Creation Graph Visualisation (Gephi) Embedding Pipelines
ProcedureDescriptionWhen to Use
apoc.export.json.allExports the entire graph (all nodes and relationships) to a JSON file or as a stream. Each element is a separate JSON object per line (JSONL format).Full graph snapshot for backup or cross-environment migration
apoc.export.json.queryRuns a Cypher query and exports its result set to JSON. The query can return nodes, relationships, paths, or scalar values.Exporting a subset of data (e.g. all Products enriched with Category) for embedding generation
apoc.export.csv.allExports all nodes and relationships as CSV files (one per label/type). Column headers are auto-generated from property keys.Producing flat-file exports for analytics tools, BI dashboards, or spreadsheet consumers
apoc.export.csv.queryExports results of a Cypher query to CSV, with query-defined column names as headers.Generating report-ready CSVs from complex graph queries
apoc.export.graphml.allExports the entire graph to GraphML XML format, compatible with tools like Gephi, yEd, and Cytoscape for visual analysis.Handing off graph data to network visualisation or analysis tools
apoc.export.cypher.allExports the graph as Cypher statements (CREATE/MERGE) that can be replayed to reconstruct the graph in another Neo4j instance.Creating portable graph snapshots; seeding test databases from production data
// Export the result of a Cypher query to a JSONL file for downstream embedding. // `stream: false` writes to disk; change to `true` to get data as a column instead. CALL apoc.export.json.query( "MATCH (p:Product)-[:BELONGS_TO]->(c:Category) RETURN p.sku AS id, p.name AS name, c.name AS category", 'file:///export/products.jsonl', {stream: false, batchSize: 1000} ) YIELD file, nodes, time RETURN file, nodes, time;

Category 03
Graph Refactoring
apoc.refactor.*

Graph Refactoring procedures let you reshape the graph structure without rewriting data. This is invaluable when evolving a data model after initial import: merging duplicate nodes, renaming labels or relationship types across the entire graph, moving properties between elements, inverting relationship directions, or extracting intermediate nodes to break direct connections into a richer multi-hop path. All operations work at scale across millions of elements through built-in batching.

Schema Evolution Deduplication Data Model Migration Relationship Reversal Node Extraction
ProcedureDescriptionWhen to Use
apoc.refactor.mergeNodesMerges a list of nodes into a single surviving node. Combines labels, properties (configurable merge strategy: overwrite, discard, combine), and re-attaches all relationships. Essential for deduplication workflows.Collapsing duplicate customer records, merging product variants
apoc.refactor.mergeRelationshipsMerges a list of parallel relationships (same type, same start/end nodes) into one, combining their properties according to a configurable merge strategy.Collapsing redundant edges after graph import
apoc.refactor.rename.labelRenames a node label across the entire graph (or a filtered subset of nodes). Adds the new label and removes the old one atomically per node.Renaming a label during schema evolution without reloading data
apoc.refactor.rename.typeChanges a relationship type to a new name across all matching relationships. The new type is created and the old one removed per relationship.Standardising relationship type names (e.g. BOUGHT → PURCHASED)
apoc.refactor.rename.nodePropertyRenames a property key on all nodes (or a filtered subset), copying the value to the new key and removing the old one.Harmonising property naming conventions across large datasets
apoc.refactor.cloneSubgraphClones a set of nodes and their connecting relationships, creating a complete independent copy of a subgraph. Useful for versioning or test fixture creation.Creating a snapshot copy of a subgraph; setting up test environments
apoc.refactor.invertRelationshipReverses the direction of a relationship, swapping its start and end nodes while preserving properties and type.Correcting incorrectly-directed edges discovered post-import
apoc.refactor.extractNodeInserts a new intermediate node along one or more existing relationships, turning A→B into A→N→B. The intermediate node gets specified labels.Enriching a direct connection with intermediate attributes (e.g. adding an Invoice node between Order and Supplier)
// Merge duplicate Customer nodes identified by matching email addresses. // apoc.refactor.mergeNodes combines all labels, props, and relationships. MATCH (c:Customer) WITH c.email AS email, collect(c) AS dupes WHERE size(dupes) > 1 CALL apoc.refactor.mergeNodes(dupes, { properties: 'combine', // combine arrays for list properties mergeRels: true // also deduplicate resulting relationships }) YIELD node RETURN count(node) AS merged;

Category 04
Graph Updates
apoc.create.*  ·  apoc.merge.*

Graph Update procedures enable dynamic, programmatic creation and modification of nodes and relationships at runtime — something that standard Cypher cannot do when label names or relationship types are not known statically at query-write time. For example, apoc.create.node accepts a list of labels computed from a variable, while apoc.merge.node merges a node dynamically using identity keys specified at runtime. These procedures are the building blocks of generic data loaders and schema-agnostic ETL frameworks.

Dynamic Labels Runtime Graph Creation Generic ETL UUID Generation Idempotent Merges
ProcedureDescriptionWhen to Use
apoc.create.nodeCreates a node with a dynamic list of labels and a map of properties. Labels can be computed from variables — impossible with plain CREATE.Generic data loaders where label names are not known at query-write time
apoc.create.relationshipCreates a relationship with a dynamic type string between two existing nodes, with a property map. Type must be a string variable.Building graph loaders that read relationship types from data files
apoc.create.setPropertySets a single property on a node or relationship where the property key is a dynamic string variable.Setting properties whose key names are determined at runtime from data
apoc.create.removePropertiesRemoves a list of property keys from a node or relationship in one call.Bulk-cleaning obsolete properties after schema migration
apoc.create.uuidGenerates a random UUID v4 string. Can be used in property assignments to ensure globally unique identifiers.Assigning stable surrogate keys to new nodes during import
apoc.merge.nodeDynamically merges a node using runtime label and identity key variables. Accepts separate property maps for ON CREATE and ON MATCH semantics.Schema-agnostic, idempotent loaders that must handle both insert and update
apoc.merge.relationshipDynamically merges a relationship with a runtime type. Accepts identity properties and separate create/match property maps.Idempotent relationship creation in generic graph loaders
apoc.nodes.deleteDeletes a collection of nodes (and optionally their relationships) in batches to avoid memory pressure from large DETACH DELETE operations.Bulk deletion of millions of nodes without heap exhaustion
// Create a node with dynamic labels — label names come from data, not code. // Standard Cypher CREATE cannot use a variable as a label name. WITH ['Product', 'Featured'] AS labels, {sku: 'P-001', name: 'Laptop Stand', price: 49.99} AS props CALL apoc.create.node(labels, props) YIELD node RETURN node;

Category 05
Path Finding & Graph Traversal
apoc.path.*

While Cypher's built-in shortestPath() and variable-length path patterns handle many traversal needs, APOC's path procedures offer far more fine-grained filtering and traversal control. You can whitelist or blacklist node labels and relationship types at each hop, restrict traversal depth, define termination conditions, and extract complete subgraphs in a single call. These procedures are essential for influence analysis, supply chain reachability, recommendation systems, and any use case requiring bounded, filtered graph exploration.

Reachability Analysis Subgraph Extraction Influence Propagation Supply Chain Analysis Recommendations
ProcedureDescriptionWhen to Use
apoc.path.expandExpands paths from one or more start nodes using a flexible filter syntax for relationship types (with direction), whitelisted and blacklisted labels, and min/max depth bounds. Returns full paths.Deep graph exploration with complex label/type filtering; supply chain reachability
apoc.path.subgraphNodesReturns all nodes reachable from the start nodes within the given depth and filter constraints, without returning the connecting relationships.Finding all nodes in a community or connected component
apoc.path.subgraphAllReturns both all reachable nodes and the relationships connecting them as a complete subgraph. Suitable for exporting or projecting a local neighbourhood.Extracting a subgraph for visualisation, export, or local analysis
apoc.path.spanningTreeComputes a spanning tree (breadth-first) from start nodes — each node appears at most once, connected by the shortest path to the root. Returns relationships only.Generating tree visualisations; computing BFS-shortest-path trees
// Expand from a Product node up to 3 hops across SUPPLIED_BY and SHIPS_TO edges. // '>' = outbound, '<' = inbound, '' = any direction. // Label filter '+Supplier|+Warehouse' whitelists only those labels. MATCH (p:Product {sku: 'ECO-42'}) CALL apoc.path.expand(p, 'SUPPLIED_BY>|SHIPS_TO>', // relationship filter '+Supplier|+Warehouse', // label whitelist 1, 3 // min depth, max depth ) YIELD path RETURN path;

Category 06
Cypher Execution & Batching
apoc.cypher.*  ·  apoc.periodic.*

This is arguably the most operationally critical APOC category. It enables dynamic Cypher execution and — crucially — periodic/batch processing of large datasets. apoc.periodic.iterate is the canonical solution to the "cannot delete 10 million nodes at once" problem: it splits an arbitrarily large dataset into fixed-size transaction batches, committing incrementally to avoid heap pressure. Background submission and scheduled Cypher jobs round out a full operational job-scheduling toolkit within the database itself.

Batch Processing Incremental Commits Dynamic Cypher Background Jobs Out-of-Memory Prevention
ProcedureDescriptionWhen to Use
apoc.periodic.iterateThe workhorse of APOC batch processing. Runs an outer "data source" query to produce rows, then applies an inner "action" query to each row in configurable transaction batches. Supports parallel execution and retries.Any large-scale write operation: importing millions of records, bulk-updating properties, batch-creating relationships
apoc.periodic.commitRuns a write Cypher statement repeatedly in separate transactions, each limited to a configurable number of changes, until the statement returns 0 affected rows. Self-terminating loop.Paginated bulk deletes or updates where the size is unknown upfront
apoc.periodic.submitSubmits a Cypher statement to run asynchronously in a background thread. Returns immediately with a job reference. Results are not streamed back.Fire-and-forget background tasks: sending notifications, updating caches
apoc.cypher.runExecutes an arbitrary Cypher read statement dynamically at runtime. The statement string and params map are passed as arguments.Meta-programming: running Cypher stored in node properties or computed from data
apoc.cypher.doItLike apoc.cypher.run but for write statements. Executes dynamic Cypher with side effects in the current transaction.Dynamic write operations in meta-driven ETL frameworks
apoc.cypher.runManyExecutes multiple Cypher statements separated by semicolons in sequence, as separate operations within one call.Running a migration script with multiple sequential steps from a single procedure call
apoc.cypher.runTimeboxedRuns a Cypher statement with a maximum execution time (ms). Aborts and returns partial results if the timeout is exceeded.Time-bounded query execution in interactive applications where partial results are acceptable
// apoc.periodic.iterate — the standard pattern for large-scale batched writes. // First query: produces rows (read-only, no write lock held across all rows). // Second query: processes each row in a separate batchSize-limited transaction. CALL apoc.periodic.iterate( "MATCH (p:Product) WHERE p.embedding IS NULL RETURN p", "SET p.needsEmbedding = true", // inner write applied per batch {batchSize: 500, parallel: false, retries: 3} ) YIELD batches, total, timeTaken, errorMessages RETURN batches, total, timeTaken, errorMessages;

Category 07
Data Structures
apoc.map.*  ·  apoc.coll.*  ·  apoc.convert.*

Cypher's native collection and map manipulation is limited. The Data Structures category provides a comprehensive functional toolkit for maps, lists, and conversion operations. apoc.map.* functions let you merge, diff, subset, and transform maps. apoc.coll.* functions extend Cypher's list operations with set algebra, sorting, grouping, zipping, and statistical operations. apoc.convert.* handles serialisation between Cypher types, JSON strings, and other formats — critical for working with properties that store serialised data.

Map Manipulation List / Collection Ops Set Algebra JSON Conversion Tree Serialisation
Procedure / FunctionDescriptionWhen to Use
apoc.map.fromPairsConverts a list of [key, value] pairs into a map object. The inverse of apoc.map.pairs.Building property maps dynamically from a list of key-value rows
apoc.map.mergeMerges two maps, with the second map's values taking precedence for overlapping keys. Returns the combined map.Combining default properties with override values during node creation
apoc.map.setKeyReturns a copy of a map with a single key set to a new value. Useful when map key names are dynamic.Adding or overriding a dynamically-named key in a map
apoc.coll.unionReturns the union of two collections (all elements from both, deduplicated). Equivalent to set union.Combining tag lists, label collections, or feature sets from two sources
apoc.coll.intersectionReturns only elements present in both collections. Equivalent to set intersection.Finding common features between two products; shared connections
apoc.coll.subtractReturns elements present in the first collection but not the second. Equivalent to set difference.Finding unique features or permissions not shared with another set
apoc.coll.toSetRemoves duplicate elements from a list, returning a deduplicated collection in the original order.Deduplicating collected values from aggregations
apoc.coll.sortNodesSorts a list of nodes by a specified property value in ascending or descending order.Sorting collected nodes before slicing for top-N results
apoc.convert.toJsonSerialises a Cypher value (node, relationship, map, list, scalar) to its JSON string representation.Storing complex structures as JSON strings in node properties
apoc.convert.fromJsonMapParses a JSON string into a Cypher map. Inverse of apoc.convert.toJson for object types.Reading JSON stored in node properties back into a traversable map
apoc.convert.toTreeConverts a list of paths into a hierarchical tree structure (nested maps). Useful for building JSON tree responses.Rendering category hierarchies, org charts, or decision trees as nested JSON
// Collect products per category and convert the result into a nested tree. MATCH path = (c:Category)<-[:BELONGS_TO]-(p:Product) WITH collect(path) AS paths CALL apoc.convert.toTree(paths) YIELD value AS tree RETURN apoc.convert.toJson(tree) AS categoryTree;

Category 08
Temporal & Date-Time
apoc.date.*  ·  apoc.temporal.*

Date and time handling is notoriously inconsistent across data sources. APOC's Temporal category provides a unified, format-aware toolkit for parsing, formatting, converting, and performing arithmetic on timestamps and Neo4j's native temporal types (Date, DateTime, LocalDateTime, Duration). The apoc.date.* functions work with Unix epoch milliseconds, while apoc.temporal.* functions work with Neo4j's ISO 8601 native temporal values — enabling clean interoperability between imported string dates and the native type system.

Date Parsing Timestamp Conversion Date Arithmetic Timezone Handling ISO 8601 Formatting
FunctionDescriptionWhen to Use
apoc.date.formatFormats a Unix timestamp (in a given unit: ms, s, m, h, d) to a date string using a Java SimpleDateFormat pattern string. Supports timezone offset.Converting stored epoch timestamps back to human-readable strings for display
apoc.date.parseParses a date string with a specified format pattern into a Unix timestamp in the specified unit. Handles custom patterns and timezones.Ingesting date strings from CSV/JSON where format varies per source
apoc.date.addAdds an amount of a time unit (ms, s, d, m, h, w) to a timestamp, returning the new timestamp in the original unit.Computing expiry dates, subscription end dates, or scheduling offsets
apoc.date.convertConverts a timestamp value from one unit to another (e.g. seconds to milliseconds, hours to minutes).Normalising timestamp units across mixed data sources
apoc.date.currentTimestampReturns the current wall-clock time as a Unix timestamp in milliseconds. Equivalent to timestamp() but within the APOC namespace for consistency.Timestamping node creation, setting TTL values, measuring elapsed time
apoc.temporal.formatFormats a Neo4j native temporal value (Date, DateTime, LocalDateTime) into a string using a format pattern.Formatting native Neo4j temporal values for export or display
apoc.temporal.toZonedTemporalParses a date/time string into a Neo4j ZonedDateTime temporal value, supporting timezone-aware datetimes.Converting imported ISO 8601 strings into Neo4j's native temporal type for indexing and arithmetic
// Parse CSV date string → epoch ms → format back for display. WITH '2024-03-15 09:30:00' AS raw WITH apoc.date.parse(raw, 'ms', 'yyyy-MM-dd HH:mm:ss') AS epoch // Add 30 days and format result WITH epoch, apoc.date.add(epoch, 'ms', 30, 'd') AS expiry RETURN apoc.date.format(epoch, 'ms', 'dd MMM yyyy') AS start, apoc.date.format(expiry, 'ms', 'dd MMM yyyy') AS expiryDate;

Category 09
Mathematical Operations
apoc.math.*  ·  apoc.number.*

APOC's Mathematical category extends Cypher's numeric capabilities with precision rounding, boundary constants, and locale-aware number parsing and formatting. These functions are particularly useful in scoring and ranking pipelines where consistent numeric formatting and rounding are required, and in ETL pipelines that consume numbers serialised in locale-specific formats (e.g. European 1.234,56 notation vs US 1,234.56).

Precision Rounding Locale-Aware Parsing Score Normalisation Numeric Formatting
FunctionDescriptionWhen to Use
apoc.math.roundRounds a floating-point number to a specified number of decimal places using a configurable rounding mode (HALF_UP, CEILING, FLOOR, etc.).Normalising similarity scores, prices, or computed metrics to a fixed precision
apoc.math.maxLongReturns the maximum value of a Java long integer (9,223,372,036,854,775,807). Useful as a sentinel value in comparisons.Setting upper bounds in range queries; sentinel values for "infinity"
apoc.math.minLongReturns the minimum Java long value. Useful as a sentinel lower bound.Setting lower bounds; initialising running minimums
apoc.number.formatFormats a number as a string using a Java DecimalFormat pattern and an optional locale tag (e.g. 'de-DE' for German notation).Generating locale-specific price strings, percentages, or large-number formatting for reports
apoc.number.parseIntParses a string to an integer using an optional radix (base). Handles hex, octal, and binary string representations.Parsing hex IDs, bitmask flags, or base-N encoded values from imported data
apoc.number.parseFloatParses a string to a floating-point number using an optional decimal format pattern, supporting locale-specific decimal separators.Parsing European-format numbers from CSV (e.g. '1.234,56' → 1234.56)
// Format a similarity score to 3 decimal places and a price to locale-specific string. WITH 0.8734521 AS rawScore, 1234.5678 AS price RETURN apoc.math.round(rawScore, 3, 'HALF_UP') AS score, apoc.number.format(price, '#,##0.00', 'en-US') AS priceUS, apoc.number.format(price, '#,##0.00', 'de-DE') AS priceDE;

Category 10
Virtual Nodes & Relationships
apoc.create.v*  ·  apoc.graph.*

Virtual Nodes and Relationships exist only in memory during query execution — they are never persisted to the database. This makes them powerful for building computed views, projection graphs, and exploratory analyses without modifying the underlying data. They are especially useful for creating aggregated "meta-nodes" that summarise communities, building temporary intermediate representations for algorithm input, or presenting a simplified view of a complex subgraph to a visualisation layer.

In-Memory Projections Graph Summaries Computed Views Algorithm Input Safe Exploration
Procedure / FunctionDescriptionWhen to Use
apoc.create.vNodeCreates a virtual node with given labels and properties. The node exists only for the duration of the current query. Appears in visualisations just like a real node.Creating in-memory aggregate nodes for graph projections; building algorithm input without writes
apoc.create.vRelationshipCreates a virtual relationship with a given type and properties between two nodes (real or virtual). Never persisted to disk.Connecting real or virtual nodes for visualisation without modifying stored relationships
apoc.create.vPatternCreates a complete virtual node-relationship-node triple in a single call. Returns the from-node, relationship, and to-node as separate yield columns.Building parameterised graph templates for repeated pattern generation
apoc.graph.fromDataAssembles a named virtual graph object from explicit lists of nodes and relationships. The graph object can be returned as a unit for visualisation.Creating a named subgraph view from a mixed set of nodes and relationships
apoc.graph.fromCypherRuns a Cypher query and assembles all returned nodes and relationships into a virtual named graph object.Packaging a complex traversal result as a labelled graph for the Neo4j Browser or Bloom
apoc.graph.fromPathsAssembles a virtual graph from a list of path objects, combining all nodes and relationships from all paths into a single deduplicated graph.Visualising the union of multiple paths as a single connected subgraph
// Create a virtual "meta-graph" showing Category → Supplier connections // without storing any new data. Useful for visualisation only. MATCH (c:Category)<-[:BELONGS_TO]-(p:Product)<-[:SUPPLIES]-(s:Supplier) WITH c, s, count(p) AS productCount CALL apoc.create.vRelationship(c, 'SOURCED_FROM', {count: productCount}, s) YIELD rel RETURN c, rel, s;

Category 11
Background Operations
apoc.periodic.*

Beyond batching, APOC provides a lightweight job scheduling and management system built directly into the database. Background jobs run Cypher statements asynchronously in separate threads, freeing the calling session immediately. Countdown timers enable self-scheduling loops. All running jobs can be inspected and cancelled at any time. While not a replacement for production schedulers (cron, Airflow, Temporal), this is invaluable for database-internal housekeeping tasks that should not block application queries.

Async Execution Job Scheduling Housekeeping TTL Expiry Non-Blocking Ops
ProcedureDescriptionWhen to Use
apoc.periodic.listReturns all currently registered and running background jobs with their names, status, delay, and associated Cypher statement.Monitoring, auditing, or debugging active background tasks
apoc.periodic.cancelCancels a named background job by its registered name. The current execution of the job (if running) is allowed to complete.Stopping a scheduled task before its next execution; emergency shutdown of runaway jobs
apoc.periodic.countdownSchedules a Cypher statement to execute repeatedly with a configurable interval (ms), decrementing a counter property on a designated node until it reaches zero, then stops.Self-terminating scheduled tasks; retry loops with a maximum attempt count
apoc.periodic.submitSubmits a Cypher statement as a named, one-shot asynchronous job. The caller receives a job reference immediately; results are not returned to the caller.Fire-and-forget operations: cache warming, index population, async notifications
// Submit a background job to pre-compute similarity scores. // The calling session is released immediately — the job runs in a background thread. CALL apoc.periodic.submit( 'compute-similarities', // job name (unique) "MATCH (a:Product),(b:Product) WHERE a <> b AND a.embedding IS NOT NULL AND b.embedding IS NOT NULL WITH a, b, apoc.math.round( reduce(d=0.0, i IN range(0,size(a.embedding)-1) | d + a.embedding[i]*b.embedding[i]), 4, 'HALF_UP') AS sim WHERE sim > 0.85 MERGE (a)-[:SIMILAR_TO {score:sim}]->(b)" ) YIELD name, submitted RETURN name, submitted;

Category 12
Database Introspection
apoc.meta.*  ·  apoc.schema.*

The Introspection category provides a live, queryable view of the graph's schema and statistics without requiring access to the Neo4j configuration files or JMX. apoc.meta.schema() returns a comprehensive JSON-like summary of all node labels, their property keys, types, and connections — effectively a runtime schema graph. apoc.meta.stats() provides label and relationship type counts. These procedures are invaluable for building schema-aware tooling, data quality dashboards, and automated documentation generators.

Utilities & Text
apoc.util.*  ·  apoc.text.*

The Utilities category is a grab-bag of essential helper functions that fill gaps left by Cypher's standard library. String manipulation functions in apoc.text.* provide fuzzy matching, case conversion, slug generation, regex group extraction, and phonetic similarity — all critical for data cleaning and entity resolution pipelines. apoc.util.* provides runtime validation, hashing, sleep, and other operational helpers that are difficult or impossible to express in pure Cypher.

// Find near-duplicate Supplier nodes using normalised Levenshtein similarity. // A score above 0.85 indicates very similar names — likely duplicates. MATCH (a:Supplier), (b:Supplier) WHERE a.id < b.id // avoid comparing a→b and b→a separately AND apoc.text.distance( toLower(a.name), toLower(b.name) ) > 0.85 RETURN a.name, b.name, apoc.math.round(apoc.text.distance(toLower(a.name), toLower(b.name)), 3, 'HALF_UP') AS similarity ORDER BY similarity DESC LIMIT 20;

Category 14
Triggers
apoc.trigger.*

APOC Triggers are Cypher-based database event hooks — reactive procedures that fire automatically when the graph is modified. Similar to SQL database triggers, they allow you to react to node or relationship creation, update, or deletion events by executing a Cypher statement in the same or a subsequent transaction. Triggers are invaluable for maintaining derived properties, audit logs, denormalised caches, and automatic relationship creation as a side effect of other writes. They must be enabled in the Neo4j configuration before use.

⚠️
Configuration Required Triggers must be enabled in neo4j.conf with apoc.trigger.enabled=true (and restarted) before any trigger procedures will work. On Neo4j 5.x, use apoc.trigger.install for cluster-aware installation.

// Install a trigger that automatically timestamps every Review node on creation. // `createdNodes` selector provides a map of newly created nodes keyed by label. CALL apoc.trigger.install( 'neo4j', // database name 'stamp-new-reviews', // unique trigger name "UNWIND $createdNodes AS n WHERE 'Review' IN labels(n) SET n.createdAt = timestamp(), n.updatedAt = timestamp()", // Cypher action ($createdNodes injected by APOC) {phase: 'after'} // fire after the creating transaction commits ) YIELD name, installed RETURN name, installed;
