A complete code walkthrough for VideoRAG — a pipeline that combines Automatic Speech Recognition, Optical Character Recognition, and Object Detection so that every spoken word, visible text, and physical object in a video becomes semantically searchable from a single query.

Executive Summary

What is VideoRAG?

VideoRAG is a production-grade Retrieval-Augmented Generation pipeline for video. Given any video file — a lecture, a product demo, a conference talk, a meeting recording — it indexes every searchable signal the video contains and lets you query it in plain language, receiving back the exact timestamped moments that are semantically relevant.

This pipeline expands the definition of "searchable signal." Where earlier implementations indexed only what was spoken, this system adds two further signal streams: what was written on screen and what was physically visible in the frame. These three streams are fused into a single embedding per chunk, meaning every search query is implicitly a search across all three dimensions simultaneously — with no extra work from the caller.

ASR
Automatic Speech Recognition
Converts the video's audio track into a precise word-level transcript with millisecond timestamps. Every spoken word — explanations, narration, off-hand remarks — becomes searchable text. Queries like "explain gradient descent" match here.
faster-whisper large-v3 · CTranslate2 · VAD filter
OCR
Optical Character Recognition
Reads all visible text from high-resolution video frames — slide titles, whiteboard equations, code on screen, UI labels, diagram annotations. Text that is shown but never spoken becomes searchable. Queries like "find the slide that says backpropagation" match here.
EasyOCR · 1280px frames · GPU-accelerated
DET
Object Detection
Identifies objects and their spatial positions in video frames — people, laptops, whiteboards, cameras, markers, diagrams. Generates a natural-language scene description per chunk. Queries like "where is someone writing on the whiteboard" match here.
YOLOv8s · COCO 80 classes · spatial labelling

How the three signals combine

For every 30-second VideoChunk the pipeline creates an enriched text string that merges all three signal outputs. This string — not just the transcript — is what gets sent to the OpenAI text embedding model. The result is that a single cosine-similarity vector search covers all three dimensions at once.

ASR
The attention mechanism computes a weighted sum of all token representations, allowing every position to directly attend to every other position in the sequence.

DET

A query like "self-attention formula with Q K V" would score highly against the OCR section of this chunk even if the speaker never uttered those exact letters. A query like "find the whiteboard segment" would score highly against the DET section. A query like "attention mechanism explanation" would score highest against the ASR transcript. One vector, three doors.

To make sense of this, you have to look at what those three-letter acronyms actually represent.

The three doors — data channels

ASR Automatic Speech Recognition the transcript
This is the text of everything the speaker said — converted from audio to words by the Whisper model, with word-level timestamps.
Why "attention mechanism explanation" fits here When a professor explains a concept they use many words and full sentences. The transcript is the richest place to find a conceptual explanation — the kind of fluid, connected language that describes how something works. No slide or visual object captures that nuance the way continuous speech does.
OCR Optical Character Recognition on-screen text
Text detected visually on screen — slide titles, whiteboard equations, code in an IDE, diagram labels, captions. Text that is shown but never spoken aloud.
Why "self-attention formula with Q K V" fits here A speaker might say "look at this math" — but the actual formula QKT is written on the slide. Even if the speaker never says "Q, K, and V" out loud, the OCR channel sees those characters on screen and matches your query against them. The formula lives in OCR, not the transcript.
DET Object Detection visual objects
The AI identifies physical objects present in video frames — not text, but things: a person, a laptop, a whiteboard, a marker, a camera. YOLOv8 runs on each frame and records what it sees, where it is, and for how long it persists.
Why "find the whiteboard segment" fits here The system isn't looking for the word "whiteboard" — it's looking for the physical object. The DET metadata records which chunks had a whiteboard present in the frame, regardless of whether it was mentioned or written on. It's a purely visual signal: "a whiteboard was visible here."

One vector, three doors — hybrid search

This phrase is a compact way of describing how the retrieval system works under the hood.

1
The vector
When you type a query, the system converts your words into a mathematical representation — a 1536-dimensional floating-point vector. This single vector encodes the semantic meaning of what you are looking for.
2
The doors
The system compares that one vector against every stored chunk embedding. Each chunk embedding was built from all three signal streams — ASR transcript, OCR on-screen text, and DET scene description — fused into a single vector. So searching one vector simultaneously reaches through all three doors at once.
3
The scoring
The score for each result is the cosine similarity between the query vector and the chunk vector. A chunk that contains matching ASR and matching OCR and matching DET will score higher than one that only matches a single signal — the fusion naturally rewards multi-signal relevance.
i
The big picture A truly smart video search doesn't just look at what was said (ASR). It looks at what was shown (OCR) and what was happening (DET) simultaneously — all in a single retrieval pass — to give the most accurate result. That is the core insight VideoRAG is built on.
i
How to read this document Sections follow data-flow order through the pipeline. ocr_engine.py and object_detector.py appear between chunking and embedding — exactly where they execute in the ingest flow.

The pipeline — coordinating all signals

VideoChunk gains four enrichment fields for the OCR and DET outputs, and ingest() adds two enrichment steps between chunking and embedding.

VideoChunk — four enrichment fields

The dataclass now carries all three signal streams. The key addition is the enriched_text property which assembles them into the structured string that the embedding model receives.

Pythonpipeline.py — VideoChunk
@dataclass
class VideoChunk:
    # ── Existing fields ─────────────────────────────────────────────
    chunk_id:    str
    video_id:    str
    start_time:  float
    end_time:    float
    transcript:  str             # ASR — what was SAID
    frame_paths: list[str]       # 336×336 JPEGs for CLIP + YOLO

    # ── New module ───────────────────────────────────────────────────
    ocr_frame_paths:   list[str]  # 1280px JPEGs for EasyOCR
    ocr_texts:         list[str]  # OCR — what was SHOWN as text
    detected_objects:  list[dict] # DET — objects with class + position
    scene_description: str        # DET — natural-language scene context

    metadata:  dict = field(default_factory=dict)

The enriched_text property — the fusion point

This property is the linchpin of the enriched-text architecture. It assembles all three signals into a structured string with labelled sections. The section headers ("On-screen text:", "Scene shows:") are intentional — they help the embedding model contextualise each segment and prevent spoken words from being confused with on-screen text.

Pythonpipeline.py — enriched_text property
@property
def enriched_text(self) -> str:
    parts = [self.transcript.strip()]       # ASR transcript always first

    if self.ocr_texts:
        ocr_joined = " | ".join(self.ocr_texts[:15])  # cap at 15 unique strings
        parts.append(f"On-screen text: {ocr_joined}")

    if self.scene_description:
        parts.append(self.scene_description)      # "Scene shows: person at center..."

    return "\n".join(parts)

backend/extractor.py
Module 03 · Extraction

Three streams, one pass

A third concurrent FFmpeg process was added for high-resolution OCR frames. All three run via asyncio.gather() — no sequential waiting.

Why two different frame resolutions?

CLIP ViT-L/14 was trained on 336×336 images — that resolution is optimal for visual embeddings and YOLO object detection (object silhouettes are clear at 336px). OCR is the opposite: a typical 24pt slide title is only ~8px tall at 336px, well below EasyOCR's reliable threshold. At 1280px that same title is ~40px tall — solidly readable. Maintaining two separate frame sets is a small storage cost for a large accuracy gain in text recognition.

Module 04 · ASR + Chunking

Transcription and dual-frame windowing

The sliding-window algorithm is unchanged. The update is mechanical: chunk() now receives and assigns two frame indexes instead of one.

The chunk() signature

Pythonchunker.py — dual frame index assignment
def chunk(self, segments, video_id,
          clip_frame_index: FrameIndex,   # 336×336 — for CLIP + YOLO
          ocr_frame_index:  FrameIndex,   # 1280px  — for EasyOCR (NEW)
          ) -> list[VideoChunk]:

    # For each window, assign both frame sets by timestamp intersection
    clip_frames = self._frames_for_window(
        clip_frame_index, chunk_start, chunk_end,
        max_frames=self.cfg.frames_per_chunk          # e.g. 4
    )
    ocr_frames = self._frames_for_window(
        ocr_frame_index, chunk_start, chunk_end,
        max_frames=self.cfg.frames_per_chunk * 2     # 8 — OCR benefits from more coverage
    )

    chunks.append(VideoChunk(
        ...,
        frame_paths     = clip_frames,
        ocr_frame_paths = ocr_frames,
        ocr_texts        = [],   # filled by OCREngine in pipeline._enrich_chunks()
        detected_objects = [],   # filled by ObjectDetector
        scene_description= "",  # filled by ObjectDetector
    ))
i
Why 2× frames for OCR? A lecture slide often appears on screen for 20–30 seconds but the content may subtly change (animations, step-by-step reveals). Using 8 frames per 30-second chunk (one every ~3.75s) vs 4 for CLIP means OCR has better coverage of incremental text changes on slides. CLIP, averaging frames for a holistic visual embedding, doesn't benefit from this extra density.

backend/ocr_engine.py
Module 05 · OCR

Reading the visible text

EasyOCR scans every high-resolution frame for text. What comes back is filtered for confidence, deduplicated across frames, and stored as a clean list of unique strings per chunk — ready to be embedded alongside the ASR transcript.

Why OCR matters for video search

In a typical 60-minute lecture, a speaker might show 40 slides, each with its own title, bullet points, equations, and code snippets. Of these, perhaps 20% are ever read aloud verbatim. The other 80% are text that the viewer reads from the screen but that never enters the ASR transcript. Without OCR, a query like "show me the slide about vanishing gradients" can only match if those words were spoken. With OCR, it matches the slide text directly — regardless of whether the speaker said it.

The data structures

Pythonocr_engine.py — TextRegion and FrameOCRResult
@dataclass
class TextRegion:
    text:       str    # Recognised string — word, phrase, or full line
    confidence: float  # EasyOCR confidence [0.0, 1.0]
    bbox:       list   # 4 corner points [[x1,y1],…,[x4,y4]] clockwise
    timestamp:  float  # Seconds from video start

    @property
    def area(self) -> float:
        # Bounding box area — used to filter tiny noise detections
        xs = [p[0] for p in self.bbox]
        ys = [p[1] for p in self.bbox]
        return (max(xs) - min(xs)) * (max(ys) - min(ys))

@dataclass
class FrameOCRResult:
    frame_path: str
    timestamp:  float
    regions:    list[TextRegion]

    @property
    def text_lines(self) -> list[str]:
        # Sort regions top-to-bottom (by bbox centroid y-coordinate)
        sorted_regions = sorted(self.regions,
                                  key=lambda r: sum(p[1] for p in r.bbox) / 4)
        return [r.text for r in sorted_regions]

ParameterValueReasoning
paragraph=FalseFalseDetects individual text lines rather than trying to merge nearby text into paragraph blocks. Better for mixed slide layouts with separate title, bullets, and code regions that EasyOCR might incorrectly merge.
min_confidence0.40Reject detections below 40% confidence. Slide text and code is typically 0.7+ confidence; noise and artefacts cluster below 0.4. Lowering this to 0.2 catches more faint text but dramatically increases noise strings in the embedding.
min_area200px²Filter bounding boxes smaller than ~14×14 pixels. Single stray characters and JPEG compression artefacts produce tiny bounding boxes that are almost never meaningful text.
lang_list["en"]English-only by default. EasyOCR supports 80+ languages — add ISO codes (e.g. "de", "fr") for multilingual content. More languages increase model init time by ~2s each.

Cross-frame deduplication — extract_chunk_texts()

A 30-second chunk might contain 30 frames, each showing the same slide. Without deduplication, "Attention is All You Need" would appear 30 times in the embedding input. The deduplication pipeline removes this redundancy while preserving all genuinely unique text.

Pythonocr_engine.py — deduplication logic
def extract_chunk_texts(self, frame_results) -> list[str]:
    # Step 1: Collect all meaningful text from all frames
    all_texts = [r.text for fr in frame_results
                        for r  in fr.regions
                        if  self._is_meaningful(r.text)]

    # Step 2: Deduplicate exact matches (normalised to lowercase)
    seen = {}
    for t in all_texts:
        normalised = " ".join(t.lower().split())
        if normalised not in seen:
            seen[normalised] = t

    # Step 3: Remove substrings  ("gradient" ⊂ "gradient descent" → keep only "gradient descent")
    filtered = [t for t in seen.values()
                if not any(t.lower() != o.lower() and
                              t.lower() in  o.lower()
                              for o in seen.values())]

    # Step 4: Sort longest first (more content) and cap at 20
    filtered.sort(key=len, reverse=True)
    return filtered[:20]
i
Why sort longest first? When the embedding model receives a list of OCR strings, the most content-rich strings should appear near the top of the "On-screen text:" section — the model's attention naturally weights earlier tokens. A slide title like "Backpropagation Through Time" carries more retrieval signal than a fragment like "Fig. 3" and should rank higher in the embedding input.

backend/object_detector.py
Module 06 · Object Detection

Seeing what's physically there

YOLOv8 scans video frames for objects, computes their spatial positions, and distils the results into a natural-language sentence that describes the visual scene of each chunk.

Why object detection matters for retrieval

ASR captures what is said. OCR captures what is written on screen. Neither captures what is physically present in the frame. A speaker demonstrating code on a laptop, a professor at a whiteboard, someone pointing at a diagram — these are visual events that have no ASR or OCR footprint, but they are exactly what a viewer means when they ask "show me where someone is writing on the board" or "find the part with the laptop demo."

The data structures

Pythonobject_detector.py — Detection and FrameDetection
@dataclass
class Detection:
    class_name: str          # COCO class label e.g. "laptop", "person"
    confidence: float        # YOLO confidence [0.0, 1.0]
    bbox_norm:  list[float]  # [cx, cy, w, h] normalised to [0,1]
    position:   str          # spatial label: "top-left", "center", etc.

@dataclass
class FrameDetection:
    frame_path: str
    timestamp:  float
    detections: list[Detection]

    @property
    def summary(self) -> str:
        # "person (left), laptop (center), whiteboard (right)"
        return ", ".join(f"{d.class_name} ({d.position})"
                         for d in self.detections)

Spatial position labels — the 3×3 grid

Each detection's normalised bounding box centroid is mapped to a human-readable spatial label using a 3×3 grid. This converts raw pixel coordinates into language that users naturally use in queries.

Pythonobject_detector.py — _position_label()
def _position_label(cx: float, cy: float) -> str:
    # Horizontal zone: 0.0–0.33 = left, 0.33–0.67 = center, 0.67–1.0 = right
    h_zone = "left"   if cx < 0.33 else "center" if cx < 0.67 else "right"
    # Vertical zone: 0.0–0.33 = top, 0.33–0.67 = middle, 0.67–1.0 = bottom
    v_zone = "top"    if cy < 0.33 else ""      if cy < 0.67 else "bottom"

    # Combine: "top-left", "center", "bottom-right", etc.
    if v_zone and h_zone != "center": return f"{v_zone}-{h_zone}"
    elif v_zone:                          return v_zone
    else:                                 return h_zone

Persistent-object filtering — the 20% rule

A single-frame detection of "cell phone" could be a genuine phone or a false positive on a shiny object. The persistence filter keeps only objects that appear in at least 20% of the chunk's frames. For a 10-frame chunk, an object must appear in at least 2 frames to be included — vastly reducing spurious detections while keeping consistently visible objects.

Pythonobject_detector.py — persistence filter in build_chunk_detections()
# Count how many frames each class appears in
class_frame_count: Counter = Counter()
for fd in frame_detections:
    for cls in {d.class_name for d in fd.detections}:
        class_frame_count[cls] += 1

# Keep only classes present in ≥ 20% of frames
min_presence = max(1, int(total_frames * 0.20))
persistent = {cls for cls, count in class_frame_count.items()
              if count >= min_presence}

Scene description generation

The scene description turns the filtered detection set into a natural-language sentence. Objects are classified into four semantic roles — people, displays, writing surfaces, and tech — and assembled in a structured sentence. This role-based ordering ensures the most retrieval-relevant information (is there a person? is there a whiteboard?) appears early in the sentence.

Pythonobject_detector.py — _describe_scene() output examples

!
COCO covers 80 common classes — not arbitrary objects YOLOv8 pre-trained on COCO can detect common objects like people, laptops, chairs, and whiteboards reliably. It cannot detect domain-specific objects like "laser pointer", "sticky note", or "specific brand of microscope." For open-vocabulary detection, replace the YOLO backbone with GroundingDINO or a CLIP-based zero-shot detector.

Module 07 · Embedding

One vector, three signals

The change to the embedder is exactly one line — but it's the line that makes everything else matter. The switch from chunk.transcript to chunk.enriched_text is what makes OCR text and scene descriptions searchable.

The one-line change

Pythonembedder.py — before and after
# Before — transcript only
texts = [c.transcript for c in chunks]

# Now — enriched_text: transcript + OCR + scene description
texts = [c.enriched_text for c in chunks]

Everything else in the embedder — the FusionProjection MLP, CLIP vision inference, asyncio.gather() concurrency, batch size, lazy model loading, zero-padded query embedding — is identical to v1. The change is purely in what text is sent to the OpenAI API.

Token length — does enriched_text exceed limits?

OpenAI's text-embedding-3-large accepts up to 8,192 tokens per input. A typical 30-second chunk produces roughly: 120–200 words of transcript (~160 tokens) + 15 OCR strings (~80 tokens) + one scene sentence (~20 tokens) = ~260 tokens total. This is well within the 8,192 limit even for the densest content. The cap of 15 OCR strings in enriched_text is a safety guard against pathological cases (e.g. a frame densely packed with small text).

i
Query embedding is unchanged embed_query() still embeds text-only with zero-padded vision. The FusionProjection MLP was designed to handle this asymmetry — it learns during training that a zero vision component means "text-only query" and still produces embeddings that land near relevant enriched-text chunk vectors.

}

These extra payload fields are returned with every search result and forwarded to the API response, allowing the frontend to display slide text, detected objects, and the scene description alongside the transcript excerpt for each result card.

HNSW settings — unchanged and why

The HNSW index configuration (m=16, ef_construct=200, on_disk=True) is identical to v1. The vectors themselves are the same dimension (1536) and the same distance metric (cosine). The only thing that changed is the quality and richness of the embeddings stored at each point — the index structure itself is agnostic to what went into producing each vector.

backend/snippet_generator.py
Module 09 · Retrieval

Cutting video snippets in milliseconds

Unchanged. The better search results produced by the three-signal embeddings automatically produce more relevant snippet cuts — no changes needed here.

The snippet generator cuts a playable MP4 clip for every search result using FFmpeg stream copy — no re-encode, sub-50ms regardless of source file length. Cuts are cached by an MD5 hash of (video_path, start_time, duration), so repeated queries for the same moment return immediately from disk. The -movflags +faststart flag moves the MP4 metadata atom to the file start so browser players can begin streaming before the full download completes.

i
Better queries → better cuts Because these embeddings now incorporate OCR text and scene descriptions, a query like "find the whiteboard formula" will retrieve a chunk that was genuinely near a whiteboard — and the snippet cut will show exactly that moment. The snippet generator doesn't know or care about the three signals; it just cuts whatever timestamps Qdrant returned. The quality improvement is upstream.

The React frontend's result cards in App.jsx display the three signal streams as distinct visual sections in the expanded card view: the transcript as flowing prose, ocr_texts as a monospace chip row (showing each detected text string as a tag), and scene_description as a small italic paragraph beneath the video player. This gives users immediate visual context for why a result was returned.
